# Разработка плагинов

Плагины MPFlow расширяют систему: свои таблицы в БД, API-роуты, middleware для обогащения данных и страницы в админке. Таблицы для плагинов создаются автоматически — не нужно писать миграции.

## Как это работает

При старте сервера:

1. Собираются entities из всех плагинов (автоматически из `src/entities/`)
2. ORM инициализируется со всеми entities (core + plugins)
3. `SchemaGenerator.updateSchema({ safe: true })` создаёт недостающие таблицы и колонки
4. Загружаются services, routes, middleware, nav и columnDocs

`safe: true` означает только аддитивные изменения — CREATE TABLE, ADD COLUMN. Без DROP.

## Convention-based auto-discovery

MPFlow использует конвенцию папок — плагин автоматически обнаруживает компоненты по структуре директорий. `plugin.ts` содержит только метаданные:

```typescript
import { definePlugin } from "../../src/server/core/plugin-loader.js"

export default definePlugin({
  name: "my-plugin",
  label: "My Plugin",
  description: "Описание вашего плагина",
})
```

Всё остальное подхватывается из конвенциональной структуры папок.

## Структура плагина

```
admin/plugins/my-plugin/
├── plugin.ts                       # Метаданные (name, label, description)
└── src/
    ├── entities/
    │   └── my-record.ts            # MikroORM entities (все экспорты — entity-классы)
    ├── services/
    │   └── my-record-service.ts    # Сервис (имя файла → DI-ключ: myRecordService)
    ├── routes/
    │   └── my-records.ts           # Hono роуты (default export, prefix: /api/my-records)
    ├── middleware/
    │   └── my-enrichment.ts        # Middleware (export config + export default handler)
    ├── workflows/
    │   └── my-sync.ts              # Логика фоновых задач
    ├── client/
    │   └── page.tsx                # React-страница в админке
    ├── nav.ts                      # Навигация (export default [{path, label}])
    └── column-docs.ts              # Документация колонок (export default [...])
```

### Конвенции именования

**Имя файла → DI-ключ** (kebab-case → camelCase):

| Файл | DI-ключ |
|------|---------|
| `my-record-service.ts` | `myRecordService` |
| `ozon-service.ts` | `ozonService` |

**Имя файла → API-префикс** (для routes):

| Файл | Префикс |
|------|---------|
| `my-records.ts` | `/api/my-records` |
| `ozon-sync.ts` | `/api/ozon-sync` |

## Пошагово: создаём плагин

### 1. Создать директорию

```bash
mkdir -p admin/plugins/my-plugin/src/{entities,services,routes}
```

### 2. Определить entity

Создайте `src/entities/my-record.ts`. Используйте декораторы MikroORM:

```typescript
import { Entity, PrimaryKey, Property, Index } from "@mikro-orm/core"
import { v4 } from "uuid"

@Entity({ tableName: "my_record" })
export class MyRecord {
  @PrimaryKey()
  id: string = v4()

  @Property()
  name!: string

  @Property({ nullable: true })
  description?: string

  @Property({ default: true })
  is_active: boolean = true

  @Property({ type: "jsonb", nullable: true })
  metadata?: Record<string, unknown>

  @Property({ defaultRaw: "now()" })
  created_at: Date = new Date()

  @Property({ defaultRaw: "now()", onUpdate: () => new Date() })
  updated_at: Date = new Date()
}
```

Таблица `my_record` будет создана автоматически при старте сервера.

Доступные типы: `string`, `number`, `boolean`, `Date`, `jsonb`, `text`, `numeric`, `bigint`. Модификаторы: `nullable`, `default`, `unique`, `index`.

### 3. Создать сервис

Создайте `src/services/my-record-service.ts`:

```typescript
import type { EntityManager } from "@mikro-orm/core"
import { MyRecord } from "../entities/my-record.js"

export class MyRecordService {
  constructor(private em: EntityManager) {}

  async list(filters: Record<string, any> = {}) {
    return this.em.find(MyRecord, filters)
  }

  async get(id: string) {
    return this.em.findOneOrFail(MyRecord, { id })
  }

  async create(data: Partial<MyRecord>) {
    const record = this.em.create(MyRecord, data as any)
    await this.em.persistAndFlush(record)
    return record
  }

  async update(id: string, data: Partial<MyRecord>) {
    const record = await this.em.findOneOrFail(MyRecord, { id })
    this.em.assign(record, data)
    await this.em.flush()
    return record
  }

  async delete(id: string) {
    const record = await this.em.findOneOrFail(MyRecord, { id })
    await this.em.removeAndFlush(record)
  }
}
```

Имя файла `my-record-service` автоматически станет DI-ключом `myRecordService`. Сервис получает `EntityManager` — request-scoped, безопасен для параллельных запросов.

### 4. Создать API-роуты

Создайте `src/routes/my-records.ts`:

```typescript
import { Hono } from "hono"
import { getUserId } from "../../../../src/server/core/auth.js"
import type { MyRecordService } from "../services/my-record-service.js"

const routes = new Hono<{ Variables: Record<string, any> }>()

routes.get("/", async (c) => {
  const userId = getUserId(c)
  const service: MyRecordService = c.get("container").resolve("myRecordService")
  const items = await service.list()
  return c.json({ items })
})

routes.post("/", async (c) => {
  const userId = getUserId(c)
  const body = await c.req.json()
  const service: MyRecordService = c.get("container").resolve("myRecordService")
  const item = await service.create(body)
  return c.json({ item }, 201)
})

export default routes
```

Имя файла `my-records` → API-префикс `/api/my-records`. Роуты регистрируются автоматически.

### 5. Написать plugin.ts

Создайте `plugin.ts` — точку входа плагина:

```typescript
import { definePlugin } from "../../src/server/core/plugin-loader.js"

export default definePlugin({
  name: "my-plugin",
  label: "My Plugin",
  description: "Описание вашего плагина",
})
```

Всё. Entities, services, routes, middleware, nav и columnDocs обнаруживаются автоматически.

### 6. Зарегистрировать плагин

Добавьте плагин в `admin/mpflow.config.ts`:

```typescript
export default defineConfig({
  // ...
  plugins: [
    { resolve: "./plugins/ozon" },
    { resolve: "./plugins/my-plugin" },  // ← ваш плагин
  ],
})
```

### 7. Запустить

```bash
cd admin
npm run dev
```

При старте в логах будет:

```
[mpflow] Collected N plugin entities
[mpflow] Applying auto-schema updates for plugin entities...
[plugin] Loading: my-plugin
[plugin] Loaded: my-plugin
```

## Дополнительные возможности

### Middleware (обогащение данных)

Middleware позволяет обогащать ответы core API. Создайте файл в `src/middleware/` с экспортом `config` и `default`:

```typescript
import type { Context, Next } from "hono"

// Конфигурация — к каким путям применяется middleware
export const config = {
  paths: [
    { path: "/api/catalog", method: "GET" },
    { path: "/api/catalog/*", method: "GET" },
  ],
}

// Обработчик — post-processing после основного хендлера
export default async function myEnrichment(c: Context, next: Next) {
  await next() // Сначала выполняется основной обработчик

  const body: any = await c.res.json()
  body.extra_data = "enriched"
  c.res = new Response(JSON.stringify(body), {
    status: c.res.status,
    headers: c.res.headers,
  })
}
```

Файлы без `export config` или без `export default` пропускаются (утилиты вроде `plugin-check.ts`).

### Навигация (nav.ts)

Для добавления пункта в сайдбар создайте `src/nav.ts`:

```typescript
export default [
  { path: "/my-plugin", label: "My Plugin" },
]
```

### Документация колонок (column-docs.ts)

Для добавления подсказок к колонкам таблиц создайте `src/column-docs.ts`:

```typescript
export default [
  {
    pageId: "warehouse",
    columnKey: "stock_total",
    pluginLabel: "My Plugin",
    description: "Описание того, что добавляет плагин в эту колонку.",
  },
]
```

### Страница в админке

Плагин может добавить страницу в клиентский SPA. Создайте React-компонент в `src/client/page.tsx` — он подхватывается через Vite glob автоматически.

### Явное определение (override)

Если конвенции не подходят, можно задать любое поле явно — оно имеет приоритет над auto-discovery:

```typescript
export default definePlugin({
  name: "my-plugin",
  label: "My Plugin",
  entities: [MyRecord],           // Явно вместо auto-discovery из src/entities/
  services: { customKey: ... },   // Явно вместо auto-discovery из src/services/
  routes: (app) => { ... },       // Явно вместо auto-discovery из src/routes/
})
```

## Контроль доступа

Пользователь может включать/выключать плагины через UI (`/plugins`). При отключении:

- API-роуты с обнаруженными `apiPrefixes` возвращают 403
- Навигация в сайдбаре скрывается
